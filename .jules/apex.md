# Apex ⚡ Performance Architect — Journal

Companion log to `.jules/sentinel.md`. Apex documents performance bottlenecks
and optimization patterns; Sentinel documents security learnings. Together
they form the project's institutional memory for cross-cutting non-feature
work.

Format mirrors Sentinel: only entries that capture a unique architectural
finding, a reusable optimization pattern, or an edge case where speed
conflicted with another property and how it was resolved. Routine micro-
optimizations belong in the PR description, not the journal.

## 2026-05-07 - Suite-Time Bottleneck: `wait`-Mocked Loops Busy-Spin Against Real-Time Deadlines
**Bottleneck:** `tests/test_thread_pool_cleanup.py::test_thread_pool_cleanup` was 25.02s — alone responsible for ~39% of the total suite wall-clock time (64.19s). The test's stated purpose is small: assert that `_collect_items` instantiates a `ThreadPoolExecutor` and uses it as a context manager. It correctly mocked `ThreadPoolExecutor`, `iter_providers`, and `wait`, but mocked `wait` to return `(set(), set())` — empty done set. The `_collect_items` loop's deadline-eviction branch (`while pending: ... if deadline is not None and now >= deadline: expired.append(future)`) reads `now` from `perf_counter()`, which advances with real wall-clock. The mocked `wait()` returned instantly but the loop kept polling `perf_counter()` until `feed_config.PROVIDER_TIMEOUT` (default 25s) had elapsed in real time. Net: a tight, CPU-bound 25-second spin in CI on every run.
**Why it matters beyond this single test:** the same anti-pattern can recur whenever a unit test mocks one half of a deadline-driven loop. If the loop reads time from a real clock (`perf_counter()`, `time.time()`, `time.monotonic()`) AND the mocked half returns instantly, the test executes the loop against real time. Future tests that mock `wait`, `select`, `poll`, or `socket.recv` while leaving deadlines intact will inherit the same spin.
**Optimization applied:** patch `feed_config.PROVIDER_TIMEOUT` to a small positive value (`0.05` — 50 ms) so the deadline arrives almost immediately. The test's actual assertions (`MockExecutor.called`, `__enter__/__exit__` triggered) are unchanged; the timeout is purely a numeric tuning of the deadline math. Result: 25.02s → 0.08s for that test, suite total 64.19s → 37.50s (~42% faster).
**Pattern for future tests:** when mocking concurrency primitives in `_collect_items`-shaped loops, also patch the timeout/deadline constant from `feed_config` to a small positive value. The deadline-eviction branch is the loop's only natural exit when the mocked wait can't deliver a "done" future. This applies equally to `wait`, `as_completed`, and any future helper that sits between the loop and the executor.

## 2026-05-07 - Data-Pipeline Bottleneck: O(n²) Regex Re-Parsing Inside `deduplicate_fuzzy`
**Bottleneck:** `src/feed/merge.py::deduplicate_fuzzy` is the final merge pass that catches near-duplicates the strict `_dedupe_items` step missed (e.g. "U1: Störung Karlsplatz" surfaced once via VOR and once via WL). The loop walks each incoming item against every already-merged item — O(n²) — and on every comparison it recomputed four expensive properties of the *existing* item: `_parse_title()` (compiled regex `_LINE_PREFIX_RE.match` + tokenize + lowercase split), `_normalize_name()` (regex strip), and `_get_tokens()` (regex split). At a typical 200-item run that's ~40,000 pair-checks × 4 regex calls = 160k redundant regex evaluations per build; the *new* item's parse was equally redundant across the inner loop's iterations. None of the existing-item parses ever change between iterations because the existing item is, by definition, immutable until the merge step replaces it.
**Why it matters beyond this single function:** the anti-pattern is "carry derived state alongside the primary data structure when the inner loop reads it". The loop authored at small-n was correct but quadratic in regex work; at 100x scale (a city-wide outage day producing ~2k disruption items pre-dedupe) the redundant parsing alone would cost a multi-second stall per build. The same shape appears whenever a fuzzy-match function does pairwise comparison and the comparison reads a derived projection of one side: the projection should travel with the item, not be recomputed.
**Optimization applied:** introduced a parallel `merged_cache: list[tuple[set[str], str, str, set[str]]]` indexed by the same position as `merged_items`. Each entry caches `(lines, name, normalized_name, tokens)` for the corresponding merged item. The inner loop now reads `merged_cache[idx]` instead of re-parsing. New helper `_compute_overlap_cache(title)` rebuilds the tuple at the *5* mutation sites (no-line append, VOR/ÖBB-direction merge, ÖBB/VOR-direction merge, peer merge, final tail append). A second helper, `_has_significant_overlap_cached(n1, n2, t1, t2)`, is the cache-aware variant of `_has_significant_overlap` — it accepts pre-computed token sets instead of re-running `_get_tokens`. Cost shifted from O(n²) regex to O(n) regex; comparison work itself stays O(n²) but now executes only set ops.
**Pattern for future code:** in any pairwise dedupe / fuzzy-merge loop, identify the projection of each operand that drives the comparison and either (a) precompute once into a parallel list when the operand is mutable in place, or (b) attach to the operand dict via a `_calculated_*` private cache key (the project's existing convention — see `_calculated_identity` in `_summarize_duplicates`). Choose (b) when the operand survives the function and the projection has reuse downstream; choose (a) when, as here, the operand is local to the function and a sibling cache list is leaner than mutating dicts.

## 2026-05-07 - Memory Hog: Defensive `copy.deepcopy(items)` in `build_feed.main`
**Bottleneck:** `src/build_feed.py::main` (formerly line ~1957) used `pre_dedupe_items = copy.deepcopy(items)` to give `_summarize_duplicates` an isolated view before `_dedupe_items` mutated the list. At a typical 200-item run with full WL/VOR/ÖBB descriptions, this peak-doubled the working set (~MBs); under the Phase 2 100x stress-day mental model (~20k items) it would drop a hundreds-of-MB transient on the heap and burn ~seconds of CPU forking dicts that `_summarize_duplicates` only reads.
**Why it matters beyond this single line:** defensive copies are a frequent reflex when one piece of code reads and a sibling mutates. They are correct but blunt: at small-n the cost is invisible, at large-n they dominate. The cheap fix is to verify the *order of operations* — `_summarize_duplicates` runs strictly before `_dedupe_items` here, so isolation is already guaranteed by sequencing; the deepcopy was paying for an invariant that ordering already provided.
**Optimization applied:** dropped the deepcopy and replaced it with `pre_dedupe_count = len(items)` (the only downstream consumer of the snapshot was the count for the log line). The reader/mutator ordering — `_summarize_duplicates(items)` then `_dedupe_items(items)` — gives `_summarize_duplicates` the pre-dedupe state at zero allocation cost. The benign `_calculated_identity` cache key that `_summarize_duplicates` writes onto each dict is deliberate (it amortises identity hashing across the run) and never affects the summary's output, so removing the copy does not regress observable behaviour.
**Pattern for future code:** before reaching for `copy.deepcopy` to isolate a reader from a mutator, check whether the call sites are sequential in time. Sequential reader-before-mutator is the cheapest possible isolation guarantee. Deepcopy is only needed when both sides hold a live reference *and* the mutator runs before / interleaved with the reader. Test-side note: assertions like `id(snapshot[0]) != id(original[0])` lock in the implementation (the deepcopy) rather than the contract (what the snapshot observes); prefer value assertions on `snapshot[0]["title"]` etc. so a future zero-copy refactor doesn't trip a regression test on an internal detail.
